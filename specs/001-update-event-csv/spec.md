# Feature Specification: Event Data Source Update

**Feature Branch**: `001-update-event-csv`  
**Created**: 2026-05-01  
**Status**: Draft  
**Input**: User description: "読み込んでいるcsvファイルをevents.csvからYosakoi-events.csvに変更したいです。目的（Why）: 正しくデータを読み込んでデータを表示したい。スコープ（What）: 新しいCSVファイルを読み込んで、正しくデータを表示できること。各よさこいイベントに推移した時のURLパスは、event_idにすること。各よさこいイベントに推移した時は、『各よさこい祭りはAIによってデータを収集させているため誤りがある可能性があります。実際の情報は公式ページでご確認ください』のような注意書きを表示すること。利用規約に追加し、本ページのデータは誤っている可能性があり、それに伴うユーザーの損失には責任を持たないことを明記する。制約（Constraints）: 正しくUI上で表示できている。"

## Clarifications

### Session 2026-05-02

- Q: `event_id` または `official_url` が欠ける不正データ行はどう扱うか？ → A: 該当行だけを一覧・詳細の対象外にする

## User Scenarios & Validation *(mandatory)*

### User Story 1 - 正しい祭り一覧を見る (Priority: P1)

利用者として、更新後のよさこいイベントデータを一覧ページで正しく見たい。そうすることで、
誤った情報ではなく、最新の掲載対象を起点に祭りを探せる。

**Why this priority**: 一覧が正しく表示されなければ、以降の詳細閲覧や導線が成立しないため。

**Independent Validation**: 一覧ページを開き、更新後のイベントデータが欠落や表示崩れなく表示され、
各カードから詳細へ移動できることを確認する。

**Acceptance Scenarios**:

1. **Given** 更新後のイベントデータが利用可能である、**When** 利用者が一覧ページを開く、
   **Then** 各イベントが正しい名称、開催地、説明、開催期間とともに表示される。
2. **Given** 一覧に複数のイベントが表示されている、**When** 利用者が任意のイベントを選択する、
   **Then** 対応するイベント詳細ページへ遷移できる。

**Responsive Checks**:

1. Mobile: 一覧カードが横スクロールなしで 1 列表示され、主要情報が切れずに読める。
2. Desktop: 一覧カードが複数列で整列し、カード間隔や文字量が崩れない。

---

### User Story 2 - 詳細ページで正しい祭り情報を確認する (Priority: P1)

利用者として、各よさこいイベントの詳細ページをイベント識別子ベースの URL で開き、注意書き付きで
内容を確認したい。そうすることで、個別ページを共有しやすくなり、情報の取り扱いにも注意できる。

**Why this priority**: 詳細ページは一覧からの遷移先であり、データの信頼性に関する案内もここで必要になるため。

**Independent Validation**: 一覧から詳細へ遷移し、URL にイベント識別子が使われていること、
AI 収集データに関する注意書きと公式ページ確認案内が表示されることを確認する。

**Acceptance Scenarios**:

1. **Given** 利用者がイベント一覧から詳細へ移動する、**When** 詳細ページが開く、
   **Then** URL パスには対象イベントの `event_id` が使われる。
2. **Given** 詳細ページが表示されている、**When** 利用者が内容を確認する、
   **Then** 「AI によってデータを収集しているため誤りの可能性がある」旨と
   「実際の情報は公式ページで確認する」旨の注意書きが表示される。
3. **Given** 詳細ページに公式情報への導線がある、**When** 利用者が正確な情報を確認したい、
   **Then** 必ず公式ページを参照できる。

**Responsive Checks**:

1. Mobile: 注意書き、基本情報、公式ページ導線が縦並びで読みやすく配置される。
2. Desktop: 基本情報と補足情報の配置が整理され、注意書きが埋もれず視認できる。

---

### User Story 3 - 利用条件上の免責を確認する (Priority: P2)

利用者として、掲載データに誤りの可能性があることと、その利用に伴う損失に対して運営側が責任を
負わないことを利用規約で確認したい。そうすることで、データの位置づけを理解したうえでサービスを利用できる。

**Why this priority**: サービス利用時の期待値調整と免責の明示が必要なため。

**Independent Validation**: 利用規約ページを開き、掲載データの誤り可能性と損失免責に関する記載が
明示されていることを確認する。

**Acceptance Scenarios**:

1. **Given** 利用者が利用規約ページを開く、**When** 免責事項を確認する、
   **Then** 本ページのデータに誤りが含まれる可能性があることが明記されている。
2. **Given** 利用者が掲載データを利用する、**When** 規約文面を確認する、
   **Then** データの誤りに起因する損失について責任を負わない旨が明記されている。

**Responsive Checks**:

1. Mobile: 免責に関する文面が折り返されても読みやすく、段落構成が崩れない。
2. Desktop: 規約文面が過度に横長にならず、重要な免責内容を読み取りやすい。

---

### Edge Cases

- 更新後のイベントデータで `event_id` または `official_url` が欠けている行は、その行のみ一覧・詳細の対象外にし、
  他の正常データは継続表示する。
- `event_id` が存在しない、重複している、または該当イベントが見つからない場合、利用者には分かりやすいエラー導線が表示される。
- `official_url` が欠けているイベントデータは不正データとして扱い、空リンクや詳細表示の継続を許可しない。
- AI 収集データの注意書きや利用規約の免責文言は、モバイル表示でも途中で切れずに読める。

## Requirements *(mandatory)*

### Must

- **FR-001**: システム MUST 更新後のよさこいイベントデータを読み込み、一覧ページに正しく表示する。
- **FR-002**: システム MUST `event_id` または `official_url` が欠ける不正データ行を掲載対象から除外し、
  他の正常データの表示は継続する。
- **FR-003**: システム MUST 一覧ページから各イベント詳細ページへ遷移できるようにする。
- **FR-004**: システム MUST 各イベント詳細ページの URL パスに `event_id` を使用する。
- **FR-005**: システム MUST 各イベント詳細ページに、AI 収集データのため誤りの可能性があることと、
  正確な情報は公式ページで確認すべきことを明記する。
- **FR-006**: システム MUST 各イベント詳細ページで `official_url` に基づく公式ページ導線を必ず表示する。
- **FR-007**: システム MUST 利用規約に、掲載データが誤っている可能性と、その利用に伴う損失への
  免責事項を明記する。
- **FR-008**: システム MUST 更新後のデータを表示しても UI が崩れず、一覧と詳細の主要情報を読める状態を保つ。

### Should

- **FR-009**: システム SHOULD `official_url` 以外のイベント情報が欠損している場合でも、表示可能な情報だけでページを成立させる。
- **FR-010**: システム SHOULD 該当イベントが見つからない場合、利用者が一覧へ戻れる分かりやすい導線を提供する。

### Could

- **FR-011**: システム COULD 注意書きを視認しやすい位置に表示し、利用者が公式情報確認の必要性を見落としにくくする。

### Key Entities *(include if feature involves data)*

- **Event Record**: 一覧と詳細に表示される、よさこい祭り 1 件分の掲載情報。
- **Event Identifier (`event_id`)**: 各イベント詳細ページの URL とイベント特定に使う一意の識別子。
- **Official URL (`official_url`)**: 各イベントに必須で紐づく、正確な情報確認先の公式ページ URL。
- **Data Accuracy Notice**: AI 収集データの誤り可能性と公式情報確認を案内する注意文。
- **Terms Disclaimer**: データ誤りの可能性と、それに伴う損失への免責を示す利用規約上の文言。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 利用者が一覧ページを開いたとき、掲載対象のイベントが欠落や表示崩れなく閲覧できる。
- **SC-002**: 利用者が一覧から任意のイベント詳細ページへ移動したとき、URL で対象イベントを識別できる。
- **SC-003**: 利用者の 100% が詳細ページ上でデータ注意書きを視認でき、公式情報確認先を見つけられる。
- **SC-004**: 利用者が利用規約ページを確認したとき、データ誤り可能性と損失免責の文言を読み取れる。

## Assumptions

- 更新後のイベントデータには、一覧表示と詳細表示に必要な基本項目が含まれている。
- 各イベントには URL に使える一意の `event_id` が定義されている。
- 各イベントには、利用者が参照できる `official_url` が必ず定義されている。
- 本機能では、データ収集方法そのものの改善や運用フロー変更は扱わない。
